OpenFGA: the authorization layer your code keeps trying to reinvent

Open any backend that’s three years old. Find the if user.role == "admin" in your handlers. Now find the second copy three files away, slightly different. The third inside a service method. The fourth, in a database query.

You’re not maintaining authorization logic. You’re scattering it.

This is the problem OpenFGA was built to solve. It’s the open-source implementation of Google’s Zanzibar paper, the system that powers Google Drive’s sharing model. Instead of asking “what role does this user have?”, you ask “is there a relationship between this user and this object that lets them do this?”.

That’s the whole shift. Once you see it, you can’t unsee it.

The core idea: relationships, not roles

Most authorization code looks like:

if user.role == "admin":
    allow()
elif user.role == "editor" and document.owner_team in user.teams:
    allow()
elif ...

The questions you actually want to answer:

  • Can Anne edit this document?
  • Can anyone in the Marketing team view this folder?
  • Which documents can Bob delete?
  • Who are the admins of this organization?

OpenFGA models all of these as relationships between objects. The data you store is a list of tuples:

user:anne   is owner   of document:roadmap
user:bob    is viewer  of document:roadmap
team:mkt    is editor  of folder:campaigns
user:carol  is member  of team:mkt

The model defines what’s possible:

model
  schema 1.1

type user

type team
  relations
    define member: [user]

type folder
  relations
    define editor: [user, team#member]
    define viewer: [user, team#member] or editor

type document
  relations
    define parent: [folder]
    define owner: [user]
    define viewer: [user] or owner or viewer from parent

Now “can Carol view document:roadmap?” becomes a question OpenFGA answers in milliseconds, even if Carol’s access comes from being a member of a team that has editor rights on the folder that contains the document.

You don’t write the traversal. You write the model.

10-minute quickstart

You need three things: the server, the CLI, and a model.

# Run OpenFGA locally
docker run -p 8080:8080 openfga/openfga run

# Install the fga CLI (needs Go 1.22+)
go install github.com/openfga/cli/cmd/fga@latest

Create a store and grab its id:

fga store create --name blog
# {"store": {"id": "01H...", "name": "blog", ...}}

export FGA_STORE_ID=01H...

Write a tiny model. Save as model.fga:

model
  schema 1.1

type user

type document
  relations
    define owner: [user]
    define viewer: [user] or owner

Apply it:

fga model write --file model.fga

Write a tuple:

fga tuple write user:anne owner document:roadmap

Ask a question:

fga query check user:anne viewer document:roadmap
# {"allowed": true}

fga query check user:bob viewer document:roadmap
# {"allowed": false}

That’s authorization, separated from your application code, queryable from any service. No more grepping handlers.

Where it actually shines

The naive use cases (admin/editor/viewer on documents) work fine with regular RBAC. OpenFGA earns its keep when you have:

  • Hierarchies. Folder inherits from organization. Document inherits from folder. Permissions cascade.
  • Group memberships. “Anyone in the Engineering team has access to this repo.” Memberships change without rewriting access lists.
  • Custom roles per resource. A document where Alice is editor, Bob is commenter, and the Marketing group is viewer.
  • Cross-tenant sharing. “Anne in Org A can comment on a doc in Org B because Org B shared a folder with Org A.”
  • Auditability. You can ask “who has access to this?” and get a real answer.

If your needs are simpler than that, RBAC is fine. If they’re more complex, you’re probably already drowning in role-explosion. OpenFGA is the way out.

What it isn’t

OpenFGA is not a free lunch. The real costs:

  • Latency. A check is a network call. Most teams co-locate OpenFGA as a sidecar to keep this low. Single-digit milliseconds is typical, but it’s not zero.
  • Data ownership. Permission tuples live in OpenFGA’s store, not your domain DB. You need to keep them in sync with your domain events (a user joining a team has to also write a tuple). Solvable, but it’s a thing.
  • Modeling discipline. Power means choice. A bad model is harder to refactor than a bad if/else chain. Spend the homework on the model upfront.
  • Operational surface. It’s one more service. Memory, storage, monitoring, upgrades.

Most of these are worth it for what you get back. None of them are free.

What a production setup looks like

A single-region OpenFGA deployment on Kubernetes, backed by Postgres for the tuple store. Co-located as a sidecar to the services that check often. A small tuple-writer consumer that listens to domain events (user joined team, doc shared, role granted) and writes the corresponding tuples. End-to-end check latency: 4-8ms p95.

Setup wasn’t free. Maintaining it has been.

Why start now

Permissions are one of the things that compound badly. A small authz mess at year one becomes “we can’t ship this feature because nobody understands who can do what anymore” at year four. The interest rate is brutal.

Pulling authz out into something modeled and queryable doesn’t have to be a rewrite. It can start as one type, one relation, one check. You grow into it.

OpenFGA is the way to make that shift. The model DSL reads cleanly, the runtime is fast, and the documentation respects your time.

Read the docs. Run the quickstart. See if the relationship model lights up something for you.

If your authz code currently lives in seventeen places, this is the way out.
