Stop pointing your tests at a real Auth0 tenant

Your service talks to Auth0. Your tests, therefore, also talk to Auth0.

That’s the moment things start to rot. The test suite needs a tenant. The tenant has rate limits. Two engineers run the suite at once and step on each other’s users. CI flakes because a token took 900ms to mint instead of 200ms. Someone hard-codes a fake JWT to skip the network, and now your signature-verification path is the one thing your tests never actually exercise.

None of this is a bug in Auth0. It’s just what happens when a test depends on a live, shared, rate-limited service it doesn’t own.

auth0-mock is the way out. It’s a self-contained service that looks and behaves like Auth0 to a calling client. You point your existing code at it instead of https://your-tenant.auth0.com, change nothing else, and your tests stop touching the network.

What it actually is

A single Go binary that speaks Auth0’s HTTP API back to you. Four things it does:

  • Mints real RS256 JWTs. Tokens are signed with an in-process key, and the matching JWKS is published at /.well-known/jwks.json. Your consumer SDK validates the signature the normal way: no InsecureSkipVerify, no fake-token kludge, no monkey-patching the one path you most want to test.
  • Covers the whole Management API. It embeds a stripped skeleton of Auth0’s OpenAPI document (~400 operations) and routes every endpoint through one generic handler. You stub a response by POSTing {method, path, response} to /admin0/expectations, and the body is checked against that operation’s schema on the spot, so a malformed stub fails at registration, not mid-test.
  • Shapes runtime state over HTTP. Custom JWT claims, per-audience permissions, the MFA-required flag, even the clock: all mutable mid-test through /admin0/*. No restart, no config-file shuffle.
  • Boots in under a second. It serves both HTTP (:8080) and HTTPS (:8443).

The point is the first bullet. Plenty of things can return canned JSON. The thing that’s annoying to fake correctly is a real signed token your real verification middleware accepts. That’s the part this gets right.

Who this is for

Anyone whose service talks to Auth0 in tests or local dev:

  • CI pipelines that need deterministic responses without burning rate limit on a shared tenant.
  • Local dev loops where you’d rather not wait on tenant latency or fight over whose test users are whose.
  • Integration suites where you want your real client library exercised end to end, not stubbed out.
  • Resilience tests for the code paths that hit /api/v2/users, /api/v2/clients, /api/v2/roles, and friends, including the unhappy ones.
  • Service-to-service flows using client_credentials, with realistic scopes and a real permissions claim shape.

It is not a production IdP, not a real RBAC engine, and not a replacement for testing against the real tenant before you ship. It’s a test double. A good one.

Quickstart

Install it. The release script downloads the goreleaser-built archive and verifies its checksum:

curl -fsSL https://raw.githubusercontent.com/sergiught/auth0-mock/main/install.sh | bash

Prefer source or Docker?

# from source
make build && ./bin/auth0-mock

# docker: pull the published image and run the binary inside it
docker run --rm -p 127.0.0.1:8080:8080 -p 127.0.0.1:8443:8443 ghcr.io/sergiught/auth0-mock:latest

Now the part that matters: proving it behaves like Auth0. Three steps: mint a token, stub a Management API response, call it with the bearer.

# 1. Mint a real signed access token
TOKEN=$(curl -s -X POST http://localhost:8080/oauth/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d 'grant_type=client_credentials&client_id=demo&client_secret=x&audience=http://localhost:8080/api/v2/' \
  | jq -r .access_token)

# 2. Stub a Management API response
curl -X POST http://localhost:8080/admin0/expectations \
  -H 'Content-Type: application/json' \
  -d '{"method":"GET","path":"/api/v2/users/auth0|123","response":{"status":200,"body":{"user_id":"auth0|123","email":"alice@example.com"}}}'

# 3. Call the stubbed endpoint with your bearer
curl http://localhost:8080/api/v2/users/auth0%7C123 \
  -H "Authorization: Bearer ${TOKEN}"
# => {"user_id":"auth0|123","email":"alice@example.com"}

That token in step 1 is real. Run it through jwt.io, or better, through your own verification middleware against the JWKS at /.well-known/jwks.json. It checks out. Your code calls the mock exactly the way it calls Auth0.

If you’d rather click than curl, run the mock and open http://localhost:8080/docs. Every endpoint is rendered as an interactive reference, and “Try it” hits the same instance.

Shaping state at runtime

This is where a mock earns its keep over a pile of static fixtures. The interesting bugs live in the variations: the admin claim that flips behavior, the missing scope that should 403, the token that expired between two calls. auth0-mock lets you drive all of that over HTTP, mid-test, no restart.

Inject a custom claim into every minted token:

curl -X PUT http://localhost:8080/admin0/claims \
  -H 'Content-Type: application/json' \
  -d '{"role":"admin","org_id":"o-42"}'

Set the permissions for an audience, so your RBAC checks have something real to read:

curl -X PUT http://localhost:8080/admin0/permissions/https://api.example.com/ \
  -H 'Content-Type: application/json' \
  -d '["read:users","write:users"]'

Force MFA on the next password grant, the half of the auth flow nobody enjoys testing:

curl -X PUT http://localhost:8080/admin0/mfa-required \
  -H 'Content-Type: application/json' \
  -d '{"required":true}'

And the one that quietly saves the most time: control the clock. Freeze it so token iat/exp are deterministic, then advance it to simulate expiry without a single sleep:

# Freeze
curl -X PUT http://localhost:8080/admin0/clock \
  -H 'Content-Type: application/json' \
  -d '{"now":"2030-01-01T00:00:00Z"}'

# Jump 7 days forward: your "token expired" path, on demand
curl -X POST http://localhost:8080/admin0/clock/advance \
  -H 'Content-Type: application/json' \
  -d '{"by":"168h"}'

Between tests, wipe everything back to zero:

curl -X POST http://localhost:8080/admin0/reset

The whole /admin0/* surface is unauthenticated on purpose, so test setup needs no token plumbing of its own. The flip side: never expose it to an untrusted network. It binds to 127.0.0.1 by default. Keep it there, inside your CI runner or dev container.

Event streams, if you need them

If your code consumes Auth0’s event streams, GET /api/v2/events is a real Server-Sent Events endpoint. You push events to it from your test, and every connected subscriber sees them live:

# subscribe in one terminal (bearer required)
curl -N -H "Authorization: Bearer $TOKEN" \
     'http://localhost:8080/api/v2/events?event_type=user.created'

# push from another
curl -X POST http://localhost:8080/admin0/events \
  -H 'Content-Type: application/json' \
  -d '{"type":"user.created","offset":"0","event":{ ... }}'

There’s a bounded replay buffer behind it, so a reconnecting subscriber can resume from Last-Event-ID and catch what it missed. Most people won’t reach for this. But if you’ve ever tried to test a consumer against a real event stream, you know why having it on tap matters.

Driving it from Go

If your tests are in Go, you don’t have to hand-marshal JSON at the /admin0 endpoints. There’s a typed client:

func TestUserLookup(t *testing.T) {
    c, err := auth0mock.NewClient("http://localhost:8080")
    if err != nil {
        t.Fatal(err)
    }
    // Reset on entry + exit, Verify all constraints at exit, correct LIFO order.
    auth0mocktest.Bracket(t, c)

    // Register a stub and demand it's hit exactly once.
    reg := auth0mocktest.MustApply(t, c.ExpectGet("/api/v2/users/auth0|alice").
        Respond(200).
        JSON(map[string]any{"user_id": "auth0|alice", "email": "alice@example.com"}))
    reg.Times(1)

    // ... code under test calls the mock the same way it calls Auth0 ...
}

Bracket(t, c) is the one-liner worth remembering: it resets the mock before your test, resets it after, and verifies every hit-count constraint at the end, in the right order, so you don’t get the “constraint silently dropped” footgun. The fluent ExpectGet(...).Respond(...).JSON(...) chain reads like the assertion it is, and .Times(n) / .AtLeast(n) / .AtMost(n) turn “this stub should fire” into something the test actually checks.

The SDK only wraps the test-fixture surface. For the Auth0 APIs themselves, you point your existing Auth0 SDK at the mock and let it do its thing.

Where it shines

The naive case (return a canned user from /api/v2/users/{id}) works with any HTTP stub. auth0-mock pulls ahead when:

  • You want the real verification path exercised. Signature, expiry, issuer, JWKS rotation. A hand-written fake token tests none of it. A real signed token tests all of it.
  • The bug is in a variation. Admin vs. non-admin, scope present vs. missing, token fresh vs. expired. Flipping those over HTTP beats maintaining a fixture per permutation.
  • You’re testing the SDK integration, not just your handler. The real client library makes the real calls. You’re testing the seam you’ll actually ship.
  • CI determinism matters. Same input, same output, no network, no rate limit, no shared-tenant contention.

If your needs are genuinely simpler than that (one endpoint, one canned response), a plain HTTP stub is fine. The moment auth itself is part of what you’re testing, faking it correctly is more work than running this.

What it isn’t

No free lunch. The honest caveats:

  • Audience is echoed, not enforced by default. The mock mints tokens with whatever audience you ask for and verifies signature, expiry, and issuer, but not that aud matches a registered API. That’s deliberate, so tests can swap audiences freely. If your downstream relies on strict aud checks, flip BEARER_REQUIRE_AUDIENCE on in your fixtures.
  • It’s a structural mock, not Auth0’s behavior. The Management API returns what you tell it to. It doesn’t model Auth0’s actual business logic, validation quirks, or every error shape. Test against a real tenant before you ship.
  • The Management skeleton is structure, not prose. It embeds the shape of Auth0’s spec (paths, methods, schemas) with Auth0’s documentation stripped out. Enough to route and validate; not a copy of their docs.
  • /admin0/* is unauthenticated. A feature for test ergonomics, a liability if exposed. Keep it on localhost.

None of these are dealbreakers for a test double. They’re the boundaries of what a test double is.

Why bother

Auth is one of those dependencies that’s invisible until it isn’t. Tests that lean on a real tenant feel fine at ten tests and miserable at five hundred: slow, flaky, contended, and quietly skipping the one path (signature verification) you most needed covered.

Moving that dependency in-process doesn’t require rewriting your auth code. You change a base URL. Your SDK keeps making the same calls. The difference is that now the thing on the other end is fast, deterministic, yours, and still hands back tokens your real verification accepts.

Run the quickstart. Point one test suite at it. See how much quieter CI gets.
